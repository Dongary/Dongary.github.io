简单的git入门-饥人谷前端路程

发表于 2018-05-30

1：注册git（https://github.com/）
2：进入 https://github.com/settings/keys
3：如果页面里已经有一些 key，就点「delete」按钮把这些 key 全删掉。如果没有，就往下看
4：点击 New SSH key，你需要输入 Title 和 Key，但是你现在没有 key，往下看
5：打开 Git Bash
6：复制并运行 rm -rf ~/.ssh/* 把现有的 ssh key 都删掉，这句命令行如果你多打一个空格，可能就要重装系统了，建议复制运行。
7：运行 ssh-keygen -t rsa -b 4096 -C “你的邮箱”，注意填写你的邮箱！
8：按回车三次
9：运行 cat ~/.ssh/id_rsa.pub，得到一串东西，完整的复制这串东西
10：回到上面第 3 步的页面，在 Title 输入「我的第一个 key」
11：在 Key 里粘贴刚刚你你复制的那串东西
12：点击 Add SSH key
13：回到 Git Bash
14：运行 ssh -T git@github.com，你可能会看到这样的提示（Are you sure you want …..）输入y
15：然后如果你看到 Permission denied (publickey). 就说明你失败了，请回到第 1 步重来，是的，回到第 1 步重来；如果你看到 Hi （your name）! You’ve successfully authenticated, but GitHub does not provide shell access. 就说明你成功了！
16：依次运行下面五句命令–

git config –global user.name (your的英文名)
git config –global user.email (your的邮箱)
git config –global push.default matching
git config –global core.quotepath false
git config –global core.editor “vim”

17:本地创建仓库–

任意地方新建XXX文件夹，gitbash进入此文件夹
git init，初始化本地仓库
git status -sb，显示当前所有文件的状态
git add xxx.xx文件路径，用来将变动加到暂存区
git commit xxx.xx -m “信息”，用来正式提交变动，提交至 .git 仓库

18：github上新建空仓库（最好与本地仓库的名字相同）
新建空仓库完成后进入的页面点击SSH按钮你会看到以下的
git remote add origin git@github.com:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx/git-demo-1.git（将本地仓库与远程仓库关联）
git push -u origin master（将本地仓推送到远程仓）

19：如果想要新建仓库，重复17后的步骤即可。

20：扩展–
git log，查看历史提交信息
git log –pretty=oneline，精简的历史信息输出
git remote -v，显示与本地仓库关联的远程仓库详细的信息
git reflog，用来记录你的每一次命令
git status, 可以让我们时刻掌握仓库当前的状态,比如文件修改但还没被提交等
git diff， 查看上次修改的具体内容
git reset –hard HEAD^，回退上次提交的版本（上一个版本就是HEAD^，上上一个版本就是HEAD^^，）
git reset –hard 版本号（列：124ea），精确回退版本，这个版本号就是下图红圈标的位置，不用全写完，一般写5位即可


版本回退是很好，不过前提是你还没有推送到远程库
git checkout – file（文件名），可以丢弃该文件在工作区的修改
git reset HEAD xx(文件名)，可以把暂存区的修改撤销掉（unstage），重新放回工作区
git rm file xx(文件名)，删除某文件
git branch，查看分支
git branch xx(分支名)，创建分支
git checkout xx(分支名)，切换分支
git checkout -b xx(分支名)，创建+切换分支
git checkout -b dev origin/dev 创建远程origin的dev分支到本地
git merge xx(分支名)，合并某分支到当前分支
git merge –no-ff -m “xxx” xx(分支名)，合并分支的时候不删除分支存在的信息
git branch -d xx(分支名)，删除分支
git branch -D xx(分支名)，强行删除某分支
git log –graph，命令可以看到分支合并图
git push origin xx(分支名) 推送分支到远程仓库的对应分支上
git stash，把当前工作现场“储藏”起来，等以后恢复现场后继续工作
需要恢复的话，有两个办法：
一是用git stash apply恢复，但是恢复后，stash内容并不删除，你需要用git stash drop来删除
另一种方式是用git stash pop，恢复的同时把stash内容也删了
git stash list，查看工作现场
git tag xx(标签名)，就可以打一个新标签
git tag，查看所有标签
git tag xx(标签名)，版本号
git tag -a xx(标签名) -m “xxx” 版本号， 为版本号多少的打上标签并添加描述xxx
git show xx(标签名)，查看标签信息
git tag -d xx(标签名)，可以删除一个本地标签
git push origin xx(标签名)，可以推送一个本地标签；
git push origin –tags，可以推送全部未推送过的本地标签
git push origin :refs/tags/xx(标签名)，可以删除一个远程标签
git config –global alias.st status，配置别名，现在git st 跟 git status的效果是一样的
21：解决合并分支的冲突问题 用git status查看冲突，必须首先解决冲突，解决冲突后，再提交，合并完成

22：在实际开发中，我们应该按照几个基本原则进行分支管理：
首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；
那在哪干活呢？干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；
你和你的小伙伴们每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了
或者修复bug时候也可以另创建分支进行修复，然后再合并
或者根据新指示添加新的功能也要另创建分支进行添加，然后再合并

23:多人协作的工作模式通常是这样：
首先，可以试图用git push origin xx(分支名)推送自己的修改；
如果推送失败，则因为远程分支比你的本地更新，需要先用git pull试图合并；
如果合并有冲突，则解决冲突，并在本地提交；
没有冲突或者解决掉冲突后，再用git push origin xx(分支名)推送就能成功！
如果git pull提示no tracking information，则说明本地分支和远程分支的链接关系没有创建，用命令git branch –set-upstream-to xx(分支名) origin/xx(分支名)。
